## 简介

LiteMark 是一个轻量级的照片参数水印工具，专为摄影爱好者设计。本指南详细说明了如何编译和验证 `lite-mark-core` 项目，包括开发构建、发布构建、单元测试、集成测试以及功能测试的具体流程。通过遵循本指南，开发者可以确保代码质量和稳定性，为项目贡献高质量的代码。

## 构建流程

### 开发构建

开发构建用于日常开发和调试，生成的可执行文件包含调试信息，便于定位问题。使用以下命令进行开发构建：

```bash
cargo build
```

该命令会在 `target/debug` 目录下生成 `litemark` 可执行文件。开发构建的编译速度较快，但生成的二进制文件体积较大，运行速度较慢。

### 发布构建

发布构建用于生产环境，生成的可执行文件经过优化，体积更小，运行速度更快。使用以下命令进行发布构建：

```bash
cargo build --release
```

该命令会在 `target/release` 目录下生成优化后的 `litemark` 可执行文件。发布构建的编译时间较长，但生成的二进制文件性能更优，适合分发和部署。

### 构建配置

项目的构建配置在 `Cargo.toml` 文件中定义，包含了项目的基本信息、依赖项和构建选项。主要依赖项包括：
- `clap`: 命令行参数解析
- `image`: 图像处理
- `rusttype`: 字体渲染
- `kamadak-exif`: EXIF 数据解析
- `serde`: JSON 序列化和反序列化

## 测试策略

### 单元测试

单元测试用于验证单个函数或模块的正确性。项目中的每个模块都包含单元测试，使用 `#[cfg(test)]` 标记。运行单元测试的命令如下：

```bash
cargo test
```

该命令会执行所有单元测试，包括 `exif_reader`、`io`、`layout` 和 `renderer` 模块的测试。测试用例覆盖了 EXIF 数据提取、文件 I/O 操作、模板变量替换和水印渲染等核心功能。

### 集成测试

集成测试用于验证多个模块协同工作的正确性。虽然项目中没有独立的集成测试文件，但 `main.rs` 中的命令行接口测试可以视为集成测试。运行集成测试的命令与单元测试相同：

```bash
cargo test
```

### 测试覆盖率

测试覆盖率是衡量测试完整性的重要指标。高覆盖率意味着大部分代码路径都经过了测试，有助于发现潜在的 bug。建议在贡献代码时，确保新增代码的测试覆盖率不低于 80%。可以使用 `cargo tarpaulin` 等工具生成测试覆盖率报告。

## 功能验证

### 使用示例图片进行测试

项目提供了多个示例图片，位于 `test_images` 目录下，用于验证功能的正确性。这些图片包括不同分辨率和方向的图像，可以全面测试水印添加功能。使用以下命令进行功能测试：

```bash
cargo run -- add -i test_images/800x600_landscape.jpg -t classic -o output.jpg --author "Photographer"
```

该命令会使用经典模板为 `800x600_landscape.jpg` 图片添加水印，并将结果保存为 `output.jpg`。通过检查输出图片，可以验证 EXIF 提取、模板渲染和图像输出的正确性。

### EXIF 提取验证

EXIF 数据提取是核心功能之一。测试时，可以使用 `exif_reader` 模块的 `extract_exif_data` 函数，验证是否能正确提取 ISO、光圈、快门速度、焦距等参数。示例代码如下：

```rust
let exif_data = litemark::exif_reader::extract_exif_data("test_images/800x600_landscape.jpg")?;
println!("Extracted EXIF data: {:?}", exif_data);
```

### 模板渲染验证

模板渲染功能通过 `layout` 模块实现。测试时，可以使用 `create_builtin_templates` 函数获取内置模板，并验证变量替换的正确性。示例代码如下：

```rust
let templates = litemark::layout::create_builtin_templates();
let classic_template = templates.iter().find(|t| t.name == "ClassicParam").unwrap();
let mut variables = std::collections::HashMap::new();
variables.insert("Author".to_string(), "John Doe".to_string());
variables.insert("ISO".to_string(), "100".to_string());
let substituted = classic_template.substitute_variables(&variables);
```

### 图像输出验证

图像输出功能通过 `io` 模块实现。测试时，可以使用 `save_image` 函数将处理后的图像保存到文件，并验证文件是否正确生成。示例代码如下：

```rust
litemark::io::save_image(&image, "output.jpg")?;
```

## 常见构建问题排查

### 依赖冲突

依赖冲突是常见的构建问题。当多个依赖项引用不同版本的同一库时，可能会导致编译失败。解决方法包括：
- 更新 `Cargo.toml` 中的依赖版本
- 使用 `cargo update` 命令更新依赖锁文件
- 检查 `Cargo.lock` 文件，确保依赖版本一致

### 平台特定问题

平台特定问题可能出现在不同操作系统上。例如，Windows 上的路径分隔符与 Unix 系统不同。解决方法包括：
- 使用 `std::path::Path` 处理路径，确保跨平台兼容性
- 在 `Cargo.toml` 中添加平台特定的依赖项
- 使用条件编译标记（如 `#[cfg(windows)]`）处理平台特定代码

### 编译错误

编译错误通常由语法错误或类型不匹配引起。解决方法包括：
- 仔细阅读编译器错误信息，定位问题代码
- 使用 `rust-analyzer` 等工具进行静态分析
- 参考 Rust 官方文档和社区资源

## 贡献代码前的测试要求

在贡献代码前，必须确保代码经过充分测试，以保证代码质量和稳定性。具体要求包括：
- 所有新增功能必须包含单元测试
- 修改现有功能时，必须更新相应的测试用例
- 确保测试覆盖率不低于 80%
- 运行 `cargo test` 确保所有测试通过
- 使用示例图片进行功能测试，验证实际效果

通过遵循这些测试要求，可以确保贡献的代码符合项目标准，减少引入 bug 的风险。
