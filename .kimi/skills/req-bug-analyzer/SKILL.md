# 需求/缺陷分析 Skill

## 核心原则

- **只读分析**：不修改任何代码文件
- **结构化输出**：使用提供的报告模板生成清晰的分析文档
- **代码驱动**：基于现有代码结构和逻辑进行分析，不做无依据的假设

## 触发时机（快速判断）

用户消息中出现以下任意关键词或模式时，**立即使用本 Skill**：

| 类型 | 关键词/模式 |
|------|------------|
| 需求分析 | "需求分析"、"分析一下需求"、"这个功能怎么做"、"怎么实现"、"评估一下影响"、"设计方案" |
| 缺陷分析 | "bug"、"为什么...不对"、"怎么回事"、"分析一下"、"根因"、"排查"、"定位"、"错误日志"、"测试失败"、"crash"、"异常" |
| 报告生成 | "写一份报告"、"输出分析文档"、"整理成文档"、"生成报告" |

## 分析流程

### 1. 明确问题

首先确认用户请求的类型：
- **需求分析**：新增功能、变更需求、优化建议
- **缺陷分析**：Bug 报告、异常现象、错误日志

收集关键信息：
- 需求/缺陷的具体描述
- 涉及的模块、页面、接口名称
- 复现步骤或期望行为
- 相关代码文件（如果用户已提供）

### 2. 代码探索

使用只读方式探索代码库：
1. **全局搜索**：通过关键词（函数名、错误信息、接口路径）定位相关文件
2. **上下文阅读**：阅读相关文件的完整逻辑，理解数据流和调用链
3. **依赖分析**：识别该需求/缺陷影响的其他模块、接口或配置
4. **边界检查**：关注异常处理、边界条件、并发场景

### 3. 分析与总结

根据分析类型，分别填充对应模板：

- **需求分析**：使用 [requirement-analysis-template.md](references/requirement-analysis-template.md)
- **缺陷分析**：使用 [bug-analysis-template.md](references/bug-analysis-template.md)

分析时重点关注：
- 现有代码中与新需求相关的逻辑
- 缺陷的根本原因（不只描述现象）
- 修改可能带来的影响范围
- 实现或修复的具体建议

### 4. 输出报告

将分析结果整理成结构化的 Markdown 报告，直接输出给用户。报告应：
- 使用对应模板作为框架
- 填充具体、可操作的结论
- 对不确定的内容明确标注"待确认"

## 模板说明

### 需求分析模板

`references/requirement-analysis-template.md` 包含以下章节：
- 需求概述（背景、目标、范围）
- 现有代码分析（相关模块、核心逻辑、依赖关系）
- 影响范围评估（代码影响、非代码影响）
- 实现建议（方案概述、关键步骤、备选方案）
- 风险与注意事项
- 验收标准

### 缺陷分析模板

`references/bug-analysis-template.md` 包含以下章节：
- 缺陷概述（现象、复现步骤、预期 vs 实际）
- 根因分析（初步定位、相关代码、根本原因）
- 影响范围评估
- 修复建议（推荐方案、代码修改点、备选方案）
- 验证与测试建议
- 预防措施

## 输出规范

1. 报告标题使用 `# 需求分析报告` 或 `# 缺陷分析报告`
2. 保持模板的章节结构，删除无内容的空章节
3. 代码引用使用文件路径 + 行号范围
4. 对关键结论加粗标注
5. 如果信息不足，在报告中明确说明缺失了哪些信息
6. 用户未指定输出目录时，将报告写入 `docs/` 目录（如 `docs/requirement-analysis-<topic>.md` 或 `docs/bug-analysis-<topic>.md`）
