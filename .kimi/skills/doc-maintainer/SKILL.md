# 文档维护工作流

## 触发条件

当用户说以下任意内容时激活：
- "检查文档"
- "维护文档"
- "确认文档"
- "文档同步"
- "看看文档有没有过期"
- "更新文档"

## 工作流概览

```
扫描文档 → 生成清单 → 委派 subagent 逐文件确认 → 汇总报告 → 更新文档
```

## 第一步：扫描文档

运行扫描脚本生成待确认清单：

```bash
python3 .kimi/skills/doc-maintainer/scripts/scan_docs.py
```

扫描结果保存在 `.kimi/doc_scan_result.json`。

## 第二步：读取扫描结果

读取 `.kimi/doc_scan_result.json` 获取待确认文件列表。按优先级（high → medium → low）逐个委派 subagent。

## 第三步：委派 subagent 确认

对每个文件，委派一个 subagent（类型：explore）进行验证。

### Subagent 指令模板

```
你是一个文档验证专家。请验证以下文档与项目代码的一致性：

文件：{file_path}
类型：{file_type}

请检查：
1. 文档描述的功能是否在代码中实现
2. 代码中的功能是否在文档中有描述
3. 配置示例、API 参数是否与实际一致
4. 链接是否有效
5. 目录结构描述是否准确

如果发现不一致：
- 详细描述不一致之处
- 提供建议的文档修改内容
- 评估严重程度（critical/warning/info）

请用中文回复，格式如下：

## 文档验证报告：{file_path}

### 状态
- [ ] 一致
- [x] 发现不一致

### 发现的问题
1. **{严重程度}**：{问题描述}
   - 文档内容：{原文}
   - 实际情况：{代码实际情况}
   - 建议修改：{修改建议}

### 建议的修改
{具体的文档修改内容，可直接用于更新文件}
```

## 第四步：汇总与更新

收集所有 subagent 的报告：

1. **如无问题**：告知用户所有文档已确认一致
2. **如有问题**：
   - 向用户汇报发现的不一致（按严重程度排序）
   - 询问用户是否确认修改
   - 用户确认后，使用 StrReplaceFile 或 WriteFile 更新文档
   - 更新时遵循最小修改原则

## 注意事项

- 每次最多并行委派 3 个 subagent（避免上下文过载）
- 优先处理 high 优先级文档
- 更新文档时保持原有风格和格式
- 不要修改 `.kimi/sessions/` 等 agent 过程文档
