# 文档维护工作流参考

## 扫描范围

- **包含**：所有 Markdown 文件（`*.md`）、JSON 配置文档、以及其他项目文档
- **排除**：
  - `.kimi/sessions/` — agent 会话记录
  - `.kimi/skills/` — skill 文件本身
  - `.trae/` — 其他 agent 过程文档
  - `archive/` — 归档文档
  - `target/` — 构建输出
  - `.git/` — Git 内部文件
  - `node_modules/`, `venv/`, `.venv/` — 依赖目录

## 文档类型与检查项

| 类型 | 优先级 | 检查项 |
|------|--------|--------|
| readme | high | 项目描述、安装步骤、快速开始、功能列表 |
| agents | high | 项目结构、技术栈版本、构建命令、开发规范 |
| architecture | high | 架构描述、模块依赖、数据流 |
| skills | high | 描述准确性、触发条件 |
| config | medium | 配置项说明、默认值、示例 |
| changelog | medium | 版本号、变更描述、日期 |
| documentation | medium | 内容时效性、链接有效性、示例代码 |
| todo | low | 完成状态、优先级 |
| plan | low | 完成进度、时间安排 |

## 子代理委派指令模板

```
你是一个文档验证专家。请验证以下文档与项目代码的一致性：

文件：{file_path}
类型：{file_type}

请检查：
1. 文档描述的功能是否在代码中实现
2. 代码中的功能是否在文档中有描述
3. 配置示例、API 参数是否与实际一致
4. 链接是否有效

如果发现不一致：
- 详细描述不一致之处
- 提供建议的文档修改内容
- 评估严重程度（critical/warning/info）

请用中文回复。
```

## 上报格式

子代理完成单个文件验证后，向主代理汇报：

```markdown
## 文档验证报告：{file_path}

### 状态
- [ ] 一致
- [x] 发现不一致

### 发现的问题
1. **{严重程度}**：{问题描述}
   - 文档内容：{原文}
   - 实际情况：{代码实际情况}
   - 建议修改：{修改建议}

### 建议的修改
{具体的文档修改内容，可直接用于更新文件}
```

## 文档更新原则

1. **最小修改**：只修改与代码不一致的部分，保持文档原有风格
2. **保持链接**：更新时注意不破坏内部链接
3. **版本敏感**：如果涉及版本变更，检查是否需要更新版本号引用
4. **格式一致**：保持原有的 Markdown 格式和缩进风格
